Visualise using dbplot

Imagine you have a huge dataset stored on Spark from which you would like to compute and show a scatter plot of two continuous numerical variables. One strategy to do this would be to collect() the data from Spark and use regular ggplot2 functions to plot it, but for large data this may not be possible: you may run out of memory on your local machine.

An alternative and better strategy is to push as much of the computation for the plot as possible to Spark. The dbplot package helps with this: it can push calculations for plotting to a remote database (Hive in this case), and bring back only the minimal data needed for a plot.

Of course, while this may be beneficial for a histogram (the database need only return a list of bins and counts), it is not so useful for scatter plots in which each data point appears as a point. The raster plot can help with this: you can think of a raster plot as being like a binned scatter plot, i.e. one in which the \(x-\) and \(y-\)coordinates have been binned. Let’s see an example for cars.

library(dbplot)

cars %>%
  dbplot_raster(x = mpg, y = hp)

Of course, this is a rather simple example in which binning is not required. The key point here though is that the binning is being done in Spark, rather than in R. `

SQL in sparklyr

Behind the scenes, sparklyr is turning your dplyr pipes into SQL commands it runs against the Hive metastore running alongside Spark. This means that we can, if we prefer, write SQL queries directly and run them in R in the way we did previously in the course

library(DBI)

query <- "SELECT AVG(hp), AVG(mpg) FROM mtcars WHERE disp > 180"
dbGetQuery(sc, query)

sparklyr native interfaces

Here ‘native’ means that these functions call Java or Scala code directly without transformation to SQL

Spark is much stricter about variable types than R (R like Python is a ‘dynamically typed’ language, but Spark is ‘statically typed’: variable types have to be defined ahead of time). In the majority of cases, native Spark functions take DoubleType inputs: this is Spark’s equivalent of an R numeric.

ft_bucketizer()

  • cut() default: include upper, exclude lower from each bin (right = TRUE), ft_bucketizer() default: include lower, exclude upper
  • ft_bucketizer() also by default includes upper and lower boundary values in uppermost bin. This is equivalent to using cut() with include.lowest = TRUE (assuming we have also set right = TRUE)
labels <- c("10 to 20", "20 to 30", "30 to 40")
mtcars %>%
  mutate(
    mpg_buckets = cut(
      mpg, 
      breaks = c(10, 20, 30, 40), 
      labels = labels,
      right = FALSE,
      include.lowest = TRUE
    )
  )

Let’s see what ft_bucketizer() returns:

cars %>%
  ft_bucketizer("mpg", "mpg_buckets", splits = c(10, 20, 30, 40)) %>%
  distinct(mpg_buckets) %>%
  collect()

It returns buckets labelled 0, 1 and 2. We should mutate these to meaningful names, but we should do the count() operation on the Spark side, as this could be an intensive computation

cars_mpg_buckets <- cars %>%
  ft_bucketizer("mpg", "mpg_buckets", splits = c(10, 20, 30, 40)) %>%
  count(mpg_buckets) %>%
  collect() %>%
  mutate(mpg_buckets = factor(mpg_buckets, labels = labels))

cars_mpg_buckets

cars_mpg_buckets <- cars %>%

ft_tokenizer() splits on spaces, but can also use ft_regex_tokenizer() to split on more complex patterns.

Note at this point that the lists in the no_stop_words column are typically smaller than the lists in the word_list column: the stop words have been removed.
